01

System Overview

FOUNDATION

What This System Does

This is a real-time financial order matching engine — the core component of any stock exchange, crypto exchange, or dark pool. It accepts buy and sell orders from traders, matches them using a price-time priority (FIFO) algorithm, executes resulting trades, and broadcasts live market data to connected clients — all with sub-millisecond latency.

Think of it as building the engine that powers NSE/BSE or Binance's order book. A buyer submits "Buy 100 shares of INFY at ₹1,500." The engine finds a seller willing to sell at ≤₹1,500, executes the trade atomically, settles the positions, and notifies both parties — in under a millisecond, continuously, at 100,000+ operations per second.

🎯 Order Ingestion

Accept, validate, and authenticate incoming orders (LIMIT, MARKET) over REST and gRPC. Apply pre-trade risk checks before any order touches the book.

⚡ Matching Algorithm

Execute price-time priority matching at the core. Single-threaded per symbol goroutine for lock-free correctness. Handle partial fills, full fills, self-match prevention.

💾 Trade Settlement

Persist executed trades to PostgreSQL atomically. Update buyer/seller balances and positions in a single DB transaction. Ensure exactly-once semantics.

📡 Real-Time Broadcast

Push order book depth updates and trade ticks to all subscribed WebSocket clients. Fan-out via Kafka → WebSocket Hub for decoupled, scalable delivery.

🔒 Auth & Risk

JWT authentication at the gateway layer. mTLS between services via Istio. Pre-trade checks: balance sufficiency, position limits, order size validation.

🔭 Observability

Prometheus metrics on every service. Grafana dashboards for throughput/latency. Jaeger distributed tracing end-to-end. Structured JSON logs for audit trail.

All 6 Microservices — Complete Definition

ARCHITECTURE
Design Philosophy

The system follows an event-driven microservices architecture with a critical constraint: the matching engine per symbol is intentionally single-threaded (one goroutine per symbol's OrderBook) to eliminate locking overhead and ensure total ordering of operations. Kafka provides the durable, replayable event bus between all services. gRPC handles synchronous service-to-service communication where a response is needed immediately.

⚠ Clarification: Why 6 Services, Not 4

One document mentioned 4 services (Gateway, Engine, Executor, Publisher). The production-grade architecture has 6 services because: (1) The WebSocket publisher is separated into its own WebSocket Hub Service instead of being embedded in the Gateway — this lets it scale independently based on connection count. (2) A dedicated User/Auth Service handles registration, login, JWT issuance/validation, keeping auth logic isolated. (3) A Risk Service intercepts every order for pre-trade validation (balance check, limits, duplicates) before it reaches Kafka — this is an industry-standard safety layer. These 3 additions transform a demo into a production-grade system.

Service 1 — API Gateway
The single entry point for all external client traffic. Handles HTTP/1.1 REST, WebSocket upgrades, JWT validation, rate limiting, and request routing. Translates REST calls into gRPC calls to internal services. Does NOT touch the order book — it is purely a routing/auth layer. Stateless — scales horizontally without coordination. This is the ONLY service exposed to the public internet.

Service 2 — Matching Engine
The performance-critical core. Maintains one OrderBook struct per trading symbol in memory. Runs one dedicated goroutine per symbol — this is the key architectural decision that eliminates lock contention. Consumes orders from Kafka (orders.submitted), executes the price-time priority matching algorithm using a B-tree (google/btree) for price levels with FIFO queues at each level, and produces Trade events to Kafka (trades.executed). Also publishes order book depth snapshots to orderbook.updates.
Port:9090 (gRPC server for streaming)
Stack Go 1.22 google/btree gRPC server kafka-go
DataIn-memory B-tree OrderBook. Never persisted to DB during operation. Rebuilt from Kafka on restart.
ScalesHorizontal by symbol shard — each instance owns a subset of symbols via Kafka partition
Service 3 — Trade Executor
Consumes the trades.executed Kafka topic and is responsible for persistent settlement: writes trade records to PostgreSQL, updates buyer/seller position and balance tables in a single atomic transaction, and emits order status updates. Uses idempotency keys (tradeID) to handle at-least-once Kafka delivery safely. Also updates Redis balance cache after successful DB writes.
Stack Go 1.22 pgx v5 go-redis v9
CommsKafka consumer → PostgreSQL TX → Redis cache write
ScalesKEDA on Kafka consumer lag — adds replicas when trades.executed backlog grows
Service 4 — WebSocket Hub
Dedicated service for real-time market data fan-out. Maintains a registry of active WebSocket connections keyed by subscribed symbol. Consumes the orderbook.updates Kafka topic and pushes depth snapshots to all clients subscribed to that symbol. Decoupled from the engine entirely — can be scaled independently. Uses goroutine-per-connection model with context cancellation for clean disconnections. Drops slow clients via backpressure (buffered send channel, capacity 256).
Port:8081 (WebSocket)
Stack Go 1.22 gorilla/websocket kafka-go
ScalesHPA on connection count or CPU — each pod handles ~10k connections (~50KB/conn)
Service 5 — User / Auth Service
Manages user registration, login, and JWT token issuance/refresh. Stores user credentials in PostgreSQL (bcrypt hashed, cost=12). Issues short-lived access tokens (15min) and long-lived refresh tokens (7d) stored in Redis. Provides a gRPC endpoint for the Gateway to validate tokens without hitting the database on every request. Handles account balance seeding for test environments.
Port:8082 (REST) / :9091 (gRPC)
Stack Go 1.22 pgx v5 jwt-go v5
CommsgRPC server (for Gateway), PostgreSQL, Redis (sessions/refresh tokens)
Service 6 — Risk Service (Pre-Trade)
Intercepts every order before it reaches Kafka/engine. Validates: sufficient available balance for the order value, position limits not exceeded, order size within symbol-specific limits, and duplicate order ID check. Reads balances from Redis cache (fast path — sub-millisecond). If Redis misses, falls back to PostgreSQL. Rejects invalid orders with specific error codes. This is an industry-standard component that prevents bad data from polluting the order book.
Stack Go 1.22 Redis read
CommsgRPC (sync, in Gateway request path, ~1ms latency)

Complete Data Flow — End to End

E2E FLOW
Example Request: "Buy 100 INFY at ₹1500"

Follow this single order from the trader's browser all the way through matching, settlement, and back to the trader's screen as a live depth update.

Flow A — Order Submission & Validation

1
Client sends REST request
Trader submits POST /api/v1/orders with JWT in Authorization header.
POST /api/v1/orders → { "symbol":"INFY","side":"BUY","type":"LIMIT","price":1500.0,"qty":100 }
2
Gateway — JWT validation + rate limiting
Gin middleware validates JWT signature via Auth Service (gRPC, cached in Redis). Extracts userID. Rate limiter checks Redis sliding window counter. If over limit → 429. If invalid JWT → 401.
Gateway → AuthSvc.ValidateToken(jwt) [gRPC, ~0.5ms] → { userID: "u123", valid: true }
3
Gateway → Risk Service pre-trade check
Does user u123 have ≥ ₹150,000 available (100 × 1500)? Order size within limits? Duplicate orderID? Risk reads from Redis balance cache for sub-millisecond checks.
Gateway → RiskSvc.CheckOrder(order) [gRPC, ~1ms] → { approved: true }
4
Gateway publishes to Kafka: orders.submitted
Serialized as Protobuf, partition key = symbol ("INFY"). Ensures all INFY orders go to the same partition. Gateway responds 202 Accepted with orderID immediately.
Kafka.Produce(topic="orders.submitted", key="INFY", value=OrderProto) → Client gets 202 {orderID:"ord-xyz"}

Flow B — Matching Logic

5
Engine consumes from orders.submitted
Consumer group engine-group. Message key "INFY" → hashed to partition 0 → this engine instance owns it. Routes to INFY goroutine via buffered channel.
engineCh["INFY"] <- order // buffered chan *Order, capacity 1000
6
Symbol goroutine runs Match()
For a BUY at ₹1500: B-tree Ascend() walks asks from lowest upward. If lowest ask ≤ 1500, match. Fill quantity FIFO from that price level. Produce []Trade. Remaining unfilled qty rests on bids side.
trades := book.Match(order) // O(log n + k) where k = number of fills
7
Engine publishes trades + book update
Produces to trades.executed (key=tradeID) and orderbook.updates (key=symbol). Both in same goroutine loop — no synchronization needed.
Kafka.Produce("trades.executed", key=trade.ID, value=TradeProto) Kafka.Produce("orderbook.updates", key="INFY", value=DepthSnapshot)

Flow C — Trade Settlement

8
Executor consumes trades.executed
Checks Redis for idempotency key trade:{tradeID}. If already processed → skip. Otherwise proceeds to DB write.
9
Atomic PostgreSQL transaction
Single transaction: INSERT trade, UPDATE balances (debit buyer, credit seller), UPDATE positions (credit buyer shares, debit seller shares), UPDATE order status. If any step fails → ROLLBACK → Kafka offset NOT committed → retry.
BEGIN; INSERT INTO trades VALUES (trade.ID, ...); UPDATE balances SET available = available - 150000 WHERE user_id = buyerID; UPDATE positions SET quantity = quantity + 100 WHERE user_id = buyerID AND symbol = 'INFY'; UPDATE orders SET filled_qty = filled_qty + 100, status = 'FILLED' WHERE id = buyOrderID; COMMIT; Redis.Set("trade:"+trade.ID, "1", 24h) // idempotency key
10
Commit Kafka offset
Only AFTER successful DB commit AND Redis key set. Guarantees: crash before offset commit → trade reprocessed on restart → idempotency key prevents double-settlement.

Flow D — Real-Time Update to Client

11
WS Hub consumes orderbook.updates
Receives depth snapshot for INFY. Looks up in-memory connection registry for all WebSocket connections subscribed to "INFY".
12
Fan-out to subscribed clients
Non-blocking send to each client's buffered sendCh. Slow clients dropped after buffer overflow.
13
Client receives live order book update
Browser gets JSON via WebSocket. UI updates the order book display in real time.
{"type":"depth","symbol":"INFY","bids":[[1500,200],[1499,500]],"asks":[[1501,100],[1502,300]],"ts":1712345678901}

PostgreSQL = source of truth for all ACID data (users, orders, trades, positions, balances). Redis = hot cache for sub-ms access (balance checks, sessions, rate limits, idempotency keys). Day 6
B-tree data structures for the order book
DSA
▼
Tasks
  • DImplement PriceLevel struct with Less() for btree ordering + FIFO queue (slice)
  • DImplement OrderBook struct: bids BTree (Descend) + asks BTree (Ascend), RWMutex, google/btree
  • DUse int64 for prices (not float64!) — scale by precision factor (1500.50 → 150050)
  • DWrite unit tests: Add, Cancel, GetBestBid, GetBestAsk — TDD from day one
  • DBenchmark with go test -bench=. -benchmem — record Add() ns/op

Why B-tree, not map? A B-tree keeps all price levels sorted — O(log n) insert/delete with natural ascending/descending iteration. For bids: Descend() walks highest first. For asks: Ascend() walks lowest first. A map has O(1) lookup but NO inherent order — you'd need a parallel sorted slice. B-tree eliminates that. Production exchanges use red-black trees (same family).

Why int64, not float64? float64 causes comparison bugs: 1500.0 == 1500.0 may fail due to IEEE 754 representation. Real exchanges use integer prices scaled by precision (1500.50 → 150050 with precision=2). This is CRITICAL — change this before anything else.

Core data structures coded, tested, benchmarked with int64 prices. You know the Add() latency in nanoseconds.

Critical Improvements & Best Practices

PRODUCTION
🔢
Use int64 for prices/quantities, NEVER float64
float64 causes comparison bugs in the matching algorithm. Real exchanges represent prices as integers scaled by precision (1500.50 → 150050). Change this on Day 6 before anything else.
CRITICAL
🌳
B-tree for price levels (already planned)
Use github.com/google/btree for O(log n) insert/delete with natural sort order. Eliminates the map + auxiliary sorted slice approach entirely.
HIGH
♻️
sync.Pool for Order and Trade structs
At 100k orders/sec, you allocate/GC 100k objects per second. sync.Pool reuses them, reducing GC pauses by 40-60% under load.
MEDIUM
📦
Protobuf for Kafka messages, not JSON
JSON: ~200 bytes per order. Protobuf: ~40 bytes. 5x smaller, 3-5x faster to serialize. Your proto files serve double duty — gRPC + Kafka.
MEDIUM
🔌
pgxpool for DB connection pooling
Use pgxpool.Pool with MaxConns: 20. Without it, under load the Executor spawns thousands of TCP connections and PostgreSQL connection limit errors cause settlement failures.
STD PRACTICE
🏥
Liveness vs Readiness probes — get them right
Liveness (/healthz): returns 200 if process alive. NEVER check external deps here. Readiness (/readyz): returns 200 only when ready to accept traffic — check Kafka connected, DB pool healthy.
STD PRACTICE
💀
Chaos engineering: kill pods and prove correctness
kubectl delete pod while load test is running. Document: how many orders reprocessed, recovery time, any double-settlements? If idempotency is correct: zero double-settlements, recovery in <10 seconds.
